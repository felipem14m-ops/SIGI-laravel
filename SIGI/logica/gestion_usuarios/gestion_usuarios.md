# Registrar Usuario

La lógica de esta funcionalidad se maneja principalmente desde el controlador Usuariocontroller.php y la vista admin/Usuarios/Formulario.blade.php.

Para realizar esta acción, el administrador autenticado interactúa con la opción "Ingresar Usuario" que se encuentra en el menú lateral izquierdo del panel de administración, dentro de la sección "Gestión de Usuarios". Este enlace está configurado para redirigir al controlador Usuariocontroller.php a través de la ruta con nombre usuarios.create, que corresponde a la URL /usuarios/crear mediante una petición GET. Esta ruta está protegida por el middleware auth, lo que garantiza que únicamente los usuarios con sesión activa puedan acceder a ella.

Al momento de recibir la petición GET, el controlador ejecuta el método create de la clase Usuariocontroller. Este método realiza una consulta a la base de datos a través del modelo Role usando el método all, que recupera todos los roles registrados en la tabla roles. Los roles obtenidos se empaquetan con la función compact y se envían a la vista Formulario.blade.php como una variable llamada roles. Esto permite que el formulario construya dinámicamente las opciones del campo de selección de roles, de manera que si en el futuro se agregan nuevos roles a la base de datos, el formulario los mostrará automáticamente sin necesidad de modificar el código de la vista.

Cuando el administrador completa los campos del formulario y hace clic en el botón "Registrar Usuario", el formulario envía los datos mediante una petición POST a la URL /usuarios, que está igualmente protegida por el middleware auth. Esta petición es recibida por el método store del controlador Usuariocontroller, que es el encargado de procesar, validar e insertar el nuevo usuario en la base de datos.

Lo primero que ejecuta el método store es una validación de los datos recibidos mediante el método validate del objeto request. El sistema verifica que el nombre no esté vacío y no supere los 255 caracteres, que el correo electrónico tenga un formato válido y que no exista previamente en la tabla users, que el rol seleccionado sea un identificador entero que realmente exista en la tabla roles, y que la contraseña tenga al menos 8 caracteres. Si alguna de estas condiciones no se cumple, Laravel interrumpe inmediatamente la ejecución del método y redirige al administrador de vuelta al formulario, mostrando los mensajes de error correspondientes junto a cada campo. El error más frecuente que puede ocurrir en este punto es el de correo electrónico duplicado, que impide que el proceso llegue a la base de datos y genera un mensaje amigable en la vista.

Una vez que todos los datos superan la validación, el controlador instancia un nuevo objeto vacío del modelo User usando la expresión new User. En este momento el registro todavía no existe en la base de datos, únicamente se crea una representation en memoria del futuro usuario. A continuación, el controlador asigna campo por campo los datos recibidos del formulario al objeto usando el método post del request para leer cada valor. La contraseña no se almacena tal como fue escrita por el administrador sino que pasa por el método make de la clase Hash, que aplica el algoritmo de cifrado Bcrypt y genera un hash seguro antes de asignarla al objeto. El campo estado se fija automáticamente con el valor activo sin requerir intervención del administrador. El campo created_by se llena con el identificador del usuario autenticado obtenido mediante Auth::id, lo que permite mantener trazabilidad de auditoría sobre qué administrador registró a cada usuario en el sistema.

Con todos los campos asignados, el controlador llama al método save del objeto usuario. En este momento Eloquent construye y ejecuta internamente la sentencia INSERT INTO users con todos los campos asignados, incluyendo los campos created_at and updated_at que son gestionados automáticamente con la fecha y hora actual del servidor. El modelo User, definido en app/Models/User.php, declara los campos permitidos para recibir datos mediante el atributo Fillable e indica que la contraseña debe ser tratada como un campo de tipo hashed en sus casts, añadiendo una capa adicional de protección sobre los datos almacenados.

Al finalizar el proceso con éxito, el controlador redirige automáticamente al administrador de vuelta al formulario de registro mediante la ruta usuarios.create, y envía un mensaje de sesión con el texto "¡Registro exitoso!". La vista detecta este mensaje usando la directiva if de sesión y muestra una alerta verde de Bootstrap en la parte superior del formulario, confirmando visualmente que el nuevo usuario fue creado correctamente en la base de datos. Si el proceso falla por cualquier razón de validación, el administrador verá los errores indicados directamente en el formulario para que pueda corregirlos y volver a intentarlo.
